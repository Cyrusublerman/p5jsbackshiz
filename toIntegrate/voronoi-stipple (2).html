VORONOI STIPPLE — documentation

A browser-based image stippling engine that converts photographs into arrangements of discrete marks (dots, symbols, or arbitrary unicode glyphs) whose size, spacing, rotation, colour and opacity encode the tonal information of the source image. The entire computation pipeline — image analysis, point distribution, iterative relaxation, and post-processing — runs in a Web Worker thread, keeping the UI responsive during heavy computation. Positions are transferred between threads as raw ArrayBuffers with zero-copy semantics.

Pipeline overview

The tool processes images through five sequential stages: image analysis, point seeding, iterative relaxation, post-processing, and rendering. Each stage is modular and configurable.

Stage 1 — Image analysis (Worker thread). The source image is resampled to the target canvas dimensions (original size × scale multiplier, capped at 3200px longest edge). Perceptual luminance is extracted per pixel using ITU-R BT.601 weights (0.299R + 0.587G + 0.114B). A Sobel operator computes horizontal and vertical gradients (Ix, Iy) using a 3×3 kernel with 0.25 normalisation. Edge magnitude is derived as √(Ix² + Iy²) and normalised to 0–1. A structure tensor is assembled from the gradient products (Jxx = Ix², Jxy = Ix·Iy, Jyy = Iy²), each component smoothed by a separable box blur (approximating Gaussian) at a user-configurable radius. The dominant orientation is extracted as 0.5·atan2(2·Jxy, Jxx − Jyy), yielding a per-pixel flow angle that follows image edges. Anisotropy magnitude is computed from the tensor eigenvalue difference. All results — edge magnitude, flow angles, flow magnitudes — are transferred back to the main thread.

Stage 2 — Density map construction (Main thread). A precomputed density map bakes together the luminance, gamma curve, inversion, and edge boost into a single Float32Array. Each pixel's density value is computed once as: d = pow(darkness, gamma) + edgeMag × edgeBoost, clamped to [0, 1]. This array is passed to the worker for seeding and relaxation, eliminating repeated per-pixel Math.pow() calls during the inner loops of relaxation (the single largest performance optimisation).

Stage 3 — Point seeding (Worker thread). An initial set of candidate points is distributed across the image. Five seeding strategies are available:

Rejection sampling — the simplest method. Random (x, y) candidates are accepted with probability proportional to the local density plus a small floor (0.02) ensuring light areas receive some points. Up to 120× the target count in attempts. Fast but produces uneven initial distributions requiring more relaxation iterations to converge.

Poisson disk sampling — variable-radius dart throwing. An average exclusion radius is estimated from the target point count and image area. For each candidate, the local exclusion radius is modulated by density: dark areas accept tighter spacing (0.2× base radius), light areas enforce wider spacing (1.0× base radius). A spatial grid accelerates the proximity test. Produces quasi-uniform initial distributions that converge in fewer Lloyd iterations. Up to 150× target count in attempts.

Blue noise threshold — a grid-stepping approach where a hash-based noise function (two interleaved sine hashes blended 70/30) is compared against the density map scaled by 1.5×. Points are placed at grid intersections with sub-cell jitter when the noise value falls below the density threshold. Produces blue-noise spectral properties without iterative refinement. Faster than Poisson but less controllable.

CDF importance sampling — builds a cumulative distribution function across all pixels (density + 0.001 floor), then samples from it via binary search. Each sample maps to a pixel position with sub-pixel jitter. Guarantees exact density-proportional distribution with no rejection waste. Best for very high point counts where rejection sampling's acceptance rate drops.

Jittered grid — divides the canvas into √(1.5 × count) cells per side, places one jittered point per cell, accepts/rejects based on local density plus a 0.05 floor. Gives stratified coverage with guaranteed spatial uniformity. The most predictable distribution but can look grid-like at low iteration counts.

Stage 4 — Relaxation (Worker thread). Points iteratively migrate toward weighted centroids of their Voronoi cells, producing the characteristic organic packing where dot spacing encodes image brightness. Four relaxation modes are available:

Lloyd's algorithm — the classic method. Each iteration: (1) build a spatial grid index, (2) for each sampled pixel, find its nearest point via expanding-ring search, (3) accumulate density-weighted centroid sums (Σ x·w, Σ y·w, Σ w) per point, (4) move each point to its weighted centroid. Pixel sampling density is controlled by the "samples" parameter (pixels per point per iteration, default 20). Higher values give more accurate Voronoi regions but cost proportionally more. The spatial grid uses cell sizes of ~1.2× the average inter-point distance for optimal bucket occupancy.

Lloyd + momentum — adds Nesterov-style acceleration. After computing the centroid, the displacement from the previous position is scaled by the momentum factor (default 0.3) and added: new_pos = centroid + momentum × (centroid − prev_pos). Converges 2–4× faster than plain Lloyd. Values above 0.5 can cause oscillation; above 0.7 is unstable for most configurations.

Jump Flood Algorithm (JFA) — approximates the Voronoi diagram on a 2× downsampled grid using the JFA propagation scheme. Starting from step = max(w,h)/2 and halving each pass, each pixel checks its 3×3 neighbourhood at the current step distance and adopts the nearest seed. After convergence, weighted centroids are accumulated from the ownership map. O(n log n) per iteration versus O(n × pixels/step²) for Lloyd. Less precise at boundaries but significantly faster for large images.

Spring / force-directed — treats points as particles. Each point experiences: (a) repulsion from neighbours within 2.5× the ideal inter-point distance, with force magnitude proportional to (ideal − actual) / ideal, scaled by 2.0; (b) attraction toward local density maxima, sampled by 6 random probes within 0.5× ideal distance, with 0.3× attraction strength. The damping parameter controls the timestep. Lower damping (0.1–0.2) gives stable but slow convergence; higher (0.6–1.0) is fast but can oscillate. Unlike Lloyd, spring relaxation doesn't compute Voronoi regions — it's a continuous force equilibrium that naturally handles boundaries.

Convergence detection monitors per-point mean squared displacement between iterations. When this energy drops below 0.01 and at least 10 iterations have elapsed, relaxation halts early. This can be disabled with the "auto-stop" checkbox to force all iterations to run.

Stage 5 — Post-processing (Worker + Main thread). After relaxation completes, three operations are applied in sequence:

Minimum distance enforcement — points are sorted by local density (darkest first). A clean spatial grid is built incrementally: each point is accepted only if no previously accepted point exists within its local minimum distance. The local minimum interpolates between min_dist (at density = 1) and max_dist (at density = 0), so dark areas pack tightly and light areas thin out. Points failing the test are discarded. This runs on the worker thread.

Capacity-constrained culling (CCVT) — optional. Points in very light areas (density < 0.15) are probabilistically removed with probability (1 − density − 0.1), balancing cell capacity across the image. A rough approximation of Balzer et al.'s full CCVT algorithm.

Noise displacement — applied on the main thread (requires p5.js noise function). Two modes: Perlin displacement adds independent x/y noise offsets; curl noise computes the curl of a scalar noise field for divergence-free (swirling) displacement that preserves point density. Both use configurable amplitude and frequency scale, with a random seed per run for variation. Points are clamped to canvas bounds after displacement.

Rendering

Every frame, the main thread computes per-point size, rotation, opacity, and colour from the current slider values (no re-run needed for these parameters).

Size mapping — size = min_size + pow(darkness, gamma) × (max_size − min_size). Gamma reshapes the tonal curve: values below 1.0 expand shadows (more points get large sizes), values above 1.0 expand highlights (more points stay small). Size jitter adds ±random variation per glyph per frame as a fraction of the computed size.

Flow field rotation — the structure tensor's dominant orientation is used to rotate glyphs. The "strength" slider blends from 0 (no rotation) to 1 (full alignment). Rotation is further modulated by local gradient magnitude (× 3, clamped to 1) so that flat regions (where orientation is undefined) don't receive arbitrary rotations. "Along edge" mode uses the tangent direction (perpendicular to gradient); "across edge" adds π/2 to follow the gradient itself.

Opacity mapping — when enabled, opacity = (1 − amount) + amount × pow(darkness, gamma). At amount = 0, all points are fully opaque. At amount = 1, light-area points fade to near-transparent. This provides additional tonal range beyond size alone.

Colour from source — samples the original image's RGB values at each point position (from the precomputed colorData Uint8Array) and renders the glyph in that colour.

Multi-glyph mixing — when multiple glyph inputs are filled, points are assigned glyphs by darkness tier. With 2 glyphs: glyph 1 for darkness 0–0.5, glyph 2 for 0.5–1.0. With 3 glyphs: thirds. The glyph input accepts any unicode character or emoji.

Flow field visualisation — when "show field" is checked, blue line segments are overlaid on the canvas at a regular grid, oriented along the flow direction with length proportional to gradient magnitude. For debugging and artistic preview.

Export

The SVG exporter writes one element per point — circles for ● glyph, <text> elements for all others — with fill colour, opacity, and rotation transform attributes matching the current render state. The SVG viewBox matches the internal canvas dimensions. Size jitter is re-randomised at export time so it won't exactly match the preview.

Controls reference

IMAGE — load: opens file picker (also triggered by tapping the canvas area before an image is loaded). scale: multiplies source dimensions by 1–4×. invert: swaps light/dark mapping.

DISTRIBUTION — dots: target point count (actual count may be lower after distance enforcement). seed: seeding strategy. edge: Sobel edge magnitude boost added to density map (0 = off, higher values concentrate points on contours). min dist / max dist: inter-point spacing bounds after relaxation; dark areas use min, light areas use max.

RELAXATION — mode: algorithm selection. iters: maximum iterations. momentum: acceleration factor for Lloyd+momentum (0 = plain Lloyd). samples: pixels sampled per point per Lloyd iteration. damping: timestep for spring mode. auto-stop: halt on convergence. ccvt: capacity-constrained culling.

TONE — gamma: power curve on darkness-to-size mapping. min/max size: glyph size range in pixels. jitter: random size variation (0–50%). opacity: darkness-to-opacity mapping strength.

GLYPH — three text inputs for unicode glyphs, assigned by darkness tier. color: sample source image colour at each point.

FLOW FIELD — strength: rotation amount (0–1). smooth: Gaussian blur radius on structure tensor (1–30). dir: along edge (tangent) or across edge (gradient). show field: overlay debug visualisation.

NOISE — perlin or curl type. amount: displacement in pixels. scale: noise frequency.

Structural parameters (dots, distances, edge boost, relaxation settings) trigger a debounced 350ms re-run of the full pipeline. Live parameters (size, gamma, opacity, jitter, rotation, glyph, colour) update every frame with no re-computation. Flow smooth recomputes only the structure tensor without re-running stippling.

FUTURE IMPROVEMENTS

Frame sequence processing and animation

The most significant extension would be video and image sequence support. The current tool processes a single static image; a sequence mode would accept multiple frames (uploaded as a folder of PNGs, an image sequence, or a video file decoded via the WebCodecs API or an ffmpeg.wasm pipeline) and stipple each frame independently or — more interestingly — with temporal coherence between frames.

Temporal coherence means that dots don't randomly jump between frames. The approach: use the previous frame's final point positions as the seed for the next frame's relaxation, then run N relaxation steps (user-configurable, perhaps 5–30) to allow the points to migrate toward the new frame's density map. With few steps, the transition is smooth and organic — dots drift like iron filings responding to a shifting magnetic field. With many steps, each frame converges more fully but the inter-frame movement is more abrupt. This creates an aesthetic spectrum from fluid morphing to discrete stipple-per-frame. The relaxation step count between frames becomes a creative parameter controlling the "viscosity" of the point cloud's response to changing content.

For looping animations (GIF, loopable MP4), the last frame's points could relax back toward the first frame's density map, creating seamless loops. For longer sequences, a sliding window approach would process frames in batches, with each batch seeded from the previous batch's final state. Memory management becomes critical here — only the current frame's density map and the point positions need to be in memory at any time; previous frames' pixel data can be discarded after density extraction.

A timeline scrubber UI would let the user preview specific frames, adjust per-frame parameters (or global parameters that apply to all frames), and set keyframes for parameter interpolation — for example, ramping the dot count up over the first 30 frames, or shifting the glyph character at specific points in the sequence. Export would produce numbered PNGs, a GIF (via gif.js), or an MP4 (via WebCodecs or ffmpeg.wasm).

Expanded glyph and character mapping

Currently the tool supports up to three glyphs assigned by darkness tier. This should expand to arbitrary-length character strings where each character in the string maps to a tonal band. Typing "·•●⬤" would create four tiers from lightest to darkest. Typing the entire ASCII printable range (" .:-=+*#%@") would approximate classic ASCII art density ramps. The mapping could be linear (equal-width bands) or perceptual (log-spaced bands matching human brightness perception). A character could also be selected by the closest match between its optical density (measured by rendering it to an offscreen canvas and computing fill ratio) and the local image density, creating a true ASCII-art mode where character selection is physically accurate rather than arbitrary.

Beyond single characters, the tool could support short strings or words as stipple elements — each point renders a word from a provided list, selected by density tier or randomly. This crosses into concrete poetry territory. Multi-character elements would need width/height measurement (via offscreen canvas TextMetrics) to avoid overlap, and the min-distance enforcement would need to account for the bounding box of each element rather than a uniform radius.

Typography controls

The current glyph rendering treats each point as an isolated mark with only size and rotation. A full typography panel would add: font-family selection (with embedded web fonts or system font enumeration), font-weight mapping (light areas render in thin weights, dark areas in bold/black — a more typographically sophisticated alternative to size scaling), italic/oblique as a density-mapped boolean or as a continuous slant angle tied to the flow field, underline/strikethrough/overline as decorations mapped to edge regions or density thresholds, letter-spacing and line-height for multi-character elements, and text-shadow or outline stroke as additional tonal modulation. The rendering pipeline would need to handle each of these as independent channels that can be mapped to different image properties (density, edges, flow angle, colour hue, saturation, etc).

A "size vs. character" toggle would let the user choose whether tonal variation is expressed through size changes (current behaviour), character substitution (all characters rendered at the same size, but different glyphs for different tones), or both. Pure character substitution at uniform size produces a distinctly different aesthetic — closer to traditional typewriter art or BBS-era ANSI art — while combined mode gives the richest tonal range.

Travelling salesman path and connectivity

An optional TSP (travelling salesman problem) solver would compute a single continuous path through all points. This transforms the stipple from a set of discrete marks into a single continuous line — a one-line drawing. The path could be rendered as a connected polyline with varying stroke width (mapped to density), or as a sequence of characters laid along the path (like text on a curve). TSP solutions can be approximated efficiently for thousands of points using nearest-neighbour heuristics, 2-opt local optimisation, or Christofides' algorithm. For artistic purposes, exact optimality doesn't matter — a "pretty good" path that avoids obvious crossings is sufficient.

Beyond TSP, other connectivity modes could include: Delaunay triangulation (connecting each point to its natural neighbours, producing a mesh whose density encodes the image), minimum spanning tree (a single tree connecting all points with minimum total edge length), k-nearest-neighbour graphs (each point connected to its k closest neighbours, producing variable-density networks), and convex hull layers (nested outlines peeling inward). Each connectivity mode creates a different visual vocabulary — meshes, webs, trees, contours — from the same point set.

Path-based rendering would also enable pen plotter output. The SVG export would contain a single <polyline> or <path> element with optimised drawing order to minimise pen-up travel. Stroke width modulation along the path would encode tonal information. For plotters, the tool could also output G-code directly.

Flow field improvements

The current flow field is derived solely from the image's luminance gradient via structure tensor. Additional flow sources would include: user-painted flow fields (click-drag on the canvas to define flow direction, interpolated across the image via RBF or diffusion), mathematical flow fields (radial, spiral, parallel, saddle, vortex — selectable from a library and composable via addition/multiplication), external vector field import (SVG path data interpreted as streamlines), and flow fields derived from image content other than luminance — for example, hue gradient (flow follows colour boundaries), saturation gradient, or a neural-network-derived saliency map.

Flow field composition would let multiple sources be blended: 50% image gradient + 30% radial from centre + 20% Perlin noise, for example. The blending would operate on the angle field using circular mean (averaging angles correctly on the unit circle) rather than naive linear interpolation, which breaks at the ±π boundary. Flow magnitude from each source would be independently controllable.

Streamline tracing would extend the flow field from a rotation source to a displacement source — points would be elongated into short curves following the flow, creating a hatching or engraving aesthetic. The streamline length could be mapped to density (long strokes in dark areas, short marks in light areas) and the streamline spacing would be controlled by the existing min/max distance parameters.

Export format expansion

Currently only SVG export is supported. The full export panel would include:

PNG / JPEG — straightforward canvas capture via toDataURL() or toBlob(). Resolution would be controllable independently of the preview canvas — the tool would render to an offscreen high-resolution canvas (up to 8192×8192) for print-quality output. JPEG quality slider for file size control.

GIF — for animated sequences (see frame sequence section), assembled via gif.js or a similar library. Frame delay, loop count, and colour quantisation (256-colour palette) would be configurable. Dithering within the GIF encoder would interact interestingly with the stipple pattern itself.

MP4 / WebM — for longer animations, encoded via the WebCodecs API (where supported) or ffmpeg.wasm as a fallback. Bitrate, codec (H.264, VP9, AV1), and framerate controls. Each frame would be rendered to an offscreen canvas and fed to the encoder.

Plain text / ASCII — when glyphs are locked to a monospace grid (see below), the output is a pure text file where each character position corresponds to a grid cell and the character is selected by density. Line breaks at row boundaries. This is traditional ASCII art generation. The text file could include ANSI colour escape codes for terminal rendering with colour.

HTML / CSS — each point rendered as an absolutely-positioned <span> element with inline styles for font-size, colour, opacity, and transform (rotation). The entire composition would be a single <div> with position: relative. This allows the stipple to be embedded in web pages as live DOM elements that can be styled, animated via CSS transitions, or manipulated with JavaScript. A CSS Grid variant would lock characters to a grid for ASCII-art-style layouts with the benefit of CSS styling (hover effects, animations, responsive scaling).

CSV / JSON — raw point data export for use in other tools. Each row: x, y, size, rotation, colour, glyph. JSON variant includes full metadata (image dimensions, all parameter values, algorithm choices) for reproducibility.

Grid-locked and line-locked modes

A "snap to grid" mode would constrain point positions to a regular monospace grid after relaxation. Each grid cell would be assigned the character whose density best matches the local image density. This bridges the gap between Voronoi stippling and traditional ASCII art — the spatial distribution is informed by Lloyd relaxation (so dense areas get more active cells) but the output is a regular text grid. Line height and column width would be configurable. The grid could be exported as plain text, HTML table, or a CSS grid layout.

A "snap to lines" variant would constrain points to horizontal baselines only, with free horizontal positioning within each line. This produces typewriter-art-style output where characters sit on consistent baselines but with irregular horizontal spacing driven by the image content. The line spacing parameter would control vertical density. Export as a text file would place characters at their approximate horizontal positions using space padding.

Background and environment controls

The current white background is hardcoded. A background panel would include: solid colour picker (with alpha), background image (a second image composited behind the stipple layer — useful for stippling a portrait over a textured paper scan), gradient backgrounds (linear, radial, conic — with stops), pattern fills (crosshatch, dots, lines — at configurable scale and angle), and transparency (PNG export with alpha channel, SVG with no background rect). Dark-on-light and light-on-dark modes would swap the fill/background relationship. A "knockout" mode would render the stipple as transparent holes in an opaque background, creating a stencil effect.

Colour space and palette controls

The current "colour from source" mode samples RGB directly. Extended colour controls would include: palette quantisation (reduce the source image to N colours before sampling, producing a screen-print aesthetic), custom palette mapping (provide a list of hex colours; each point snaps to the nearest palette colour in CIELAB space), hue rotation and saturation adjustment applied to the sampled colours, separate colour channels rendered as separate stipple layers (CMYK separation — four overlaid stipple passes in cyan, magenta, yellow, and black, each driven by its respective channel's density, producing a halftone-style colour print effect), and monochrome tinting (all points rendered in a single user-selected colour with density mapped to opacity or size).

Interactivity and real-time modes

A webcam input mode would capture frames from the device camera and stipple them in real-time. With aggressive downsampling (200×150 luminance map), reduced point counts (~2000), and minimal relaxation (3–5 iterations of Lloyd per frame, seeded from the previous frame's positions), real-time stippling at 15–20fps is feasible on modern hardware. The temporal coherence approach described in the animation section applies directly — points drift organically in response to camera movement, creating a living stipple portrait.

Interactive point editing would let the user click to add, remove, or drag individual points. A brush tool could paint density modifications onto the density map (boost or suppress points in regions), and a "freeze" tool could lock selected points in place during subsequent relaxation iterations. An undo/redo stack would track all edits.

Multi-layer composition

Multiple stipple layers could be composited, each with independent settings. For example: a base layer of small dense circles for overall tone, a second layer of larger triangles aligned to the flow field for texture, and a third layer of coloured emoji for highlights. Each layer would have its own point set, glyph, size range, and blending mode. Layers could be reordered, soloed, muted, and exported individually or merged. This approaches a full compositing engine built on stippling as the atomic operation.

Performance and scalability

For point counts above 30,000, the spatial grid's expanding-ring search becomes a bottleneck. A k-d tree with median-split construction would provide O(log n) nearest-neighbour queries. For the Lloyd centroid computation, the pixel-sampling approach could be replaced with GPU-accelerated Voronoi diagram generation using jump flood on a WebGL framebuffer — each pixel would be coloured by its nearest seed's index (encoded in the colour channels), and the centroid would be computed by reading back the framebuffer and accumulating per-seed weighted sums. This would shift the O(pixels) work to the GPU, leaving only the O(n) centroid update on the CPU. With WebGPU (where available), compute shaders could run the entire relaxation loop on the GPU, handling 100,000+ points at interactive rates.

OffscreenCanvas in dedicated workers would allow rendering to happen off the main thread, eliminating frame drops during relaxation. SharedArrayBuffer (where available with correct COOP/COEP headers) would allow the point position array to be shared between the relaxation worker and the render worker without copying, further reducing memory bandwidth.

For extremely large outputs (poster-size prints at 300dpi, which may involve 50,000–500,000 points), the rendering could be tiled — the canvas split into rectangular regions, each rendered independently and stitched into the final output. This avoids exceeding the browser's maximum canvas size limits (typically 16384×16384 or 32768×32768 depending on GPU).

Preset system and parameter serialisation

A preset system would save and restore complete parameter states as JSON objects. Built-in presets would cover common use cases: "classic stipple" (circles, Poisson seed, high iteration count, no colour), "ASCII art" (full character ramp, grid-locked, text export), "hatching" (elongated marks, strong flow field, along-edge direction), "pointillist" (colour from source, size jitter, medium dots), "one-line drawing" (TSP path, continuous polyline), "halftone" (regular grid, circles only, size-mapped). User presets could be saved to localStorage and exported/imported as JSON files. A randomise button would generate random parameter combinations within sensible bounds for creative exploration.

Accessibility and documentation

Parameter tooltips on hover/long-press would explain each control in one sentence. A guided tutorial mode would walk new users through loading an image and adjusting key parameters with before/after previews. The SVG export could include <title> and <desc> elements with alt-text describing the source image content (user-provided or auto-generated via a vision API). Keyboard shortcuts for common actions (reset, export, toggle preview) would improve workflow speed. A comparison mode would render two parameter sets side-by-side for A/B evaluation.
